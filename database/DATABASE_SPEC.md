# Yelena Inventory Database Specification

## Overview

Yelena Inventory uses Supabase Free with PostgreSQL as its shared backend
foundation. The system is single tenant and server first. PostgreSQL is the
central source of truth. The schema uses UUID primary keys generated by
`pgcrypto`.

Business records are normally deactivated with `is_active = false`. Inventory
counts and audit records are historical events and are retained.

## Extension

### `pgcrypto`

Provides `gen_random_uuid()` for UUID primary key defaults.

## Tables

### `branches`

Stores business branches.

| Column | Type | Rules |
|---|---|---|
| `id` | `uuid` | Primary key; defaults to `gen_random_uuid()` |
| `name` | `text` | Required and unique |
| `is_active` | `boolean` | Required; defaults to `true` |
| `created_at` | `timestamptz` | Required; defaults to `now()` |
| `updated_at` | `timestamptz` | Required; defaults to `now()` and maintained by trigger |
| `branch_code` | `text` | Required, unique, non-empty, and contains no whitespace |

Constraints:

- `chk_branches_name_not_empty`
- `chk_branches_branch_code_not_empty`
- `chk_branches_branch_code_no_spaces`
- `uq_branches_name`
- `uq_branches_branch_code`

There is no `manager_id`. Manager assignment is intentionally postponed until
users and roles are designed.

### `employees`

Stores employees independently from branch membership.

| Column | Type | Rules |
|---|---|---|
| `id` | `uuid` | Primary key; defaults to `gen_random_uuid()` |
| `name` | `text` | Required and non-empty; not unique |
| `is_active` | `boolean` | Required; defaults to `true` |
| `created_at` | `timestamptz` | Required; defaults to `now()` |
| `phone` | `text` | Required, unique, non-empty, and contains no whitespace |
| `updated_at` | `timestamptz` | Required; defaults to `now()` and maintained by trigger |
| `employee_code` | `text` | Required, unique, non-empty, and contains no whitespace |

Constraints:

- `chk_employees_name_not_empty`
- `chk_employees_phone_not_empty`
- `chk_employees_phone_no_spaces`
- `chk_employees_employee_code_not_empty`
- `chk_employees_employee_code_no_spaces`
- `uq_employees_phone`
- `uq_employees_employee_code`

Two employees may have the same name. Phone numbers are expected to move
toward E.164 formatting, such as `+972501234567`; detailed format validation is
currently an application responsibility.

### `employee_branches`

Represents the many-to-many relationship between employees and branches.

| Column | Type | Rules |
|---|---|---|
| `id` | `uuid` | Primary key; defaults to `gen_random_uuid()` |
| `employee_id` | `uuid` | Required; references `employees(id)` |
| `branch_id` | `uuid` | Required; references `branches(id)` |
| `is_active` | `boolean` | Required; defaults to `true` |
| `created_at` | `timestamptz` | Required; defaults to `now()` |
| `updated_at` | `timestamptz` | Required; defaults to `now()` and maintained by trigger |

Constraint:

- `uq_employee_branch` on (`employee_id`, `branch_id`)

Future branch-specific roles belong to this relationship, but no role columns
exist in the current schema.

Indexes:

- `idx_employee_branches_employee` on (`employee_id`)
- `idx_employee_branches_branch` on (`branch_id`)

### `products`

Stores products identified by immutable barcodes.

| Column | Type | Rules |
|---|---|---|
| `id` | `uuid` | Primary key; defaults to `gen_random_uuid()` |
| `barcode` | `text` | Required, unique, non-empty, and contains no whitespace |
| `name` | `text` | Optional |
| `notes` | `text` | Optional |
| `image_path` | `text` | Optional path to the product's single image |
| `is_active` | `boolean` | Required; defaults to `true` |
| `created_at` | `timestamptz` | Required; defaults to `now()` |
| `updated_at` | `timestamptz` | Required; defaults to `now()` and maintained by trigger |

Constraints:

- Unique constraint declared on `barcode`
- `chk_products_barcode_not_empty`
- `chk_products_barcode_no_spaces`

Indexes:

- The unique barcode constraint supplies the barcode index. No duplicate
  barcode index is created.
- `idx_products_active` on (`is_active`)

One product equals one barcode. A barcode is immutable within this application;
a different barcode represents a different product. Each product has at most
one image, stored directly through `image_path`; there is no
`product_images` table.

### `inventory_counts`

Stores every inventory count as an event.

| Column | Type | Rules |
|---|---|---|
| `id` | `uuid` | Primary key; defaults to `gen_random_uuid()` |
| `product_id` | `uuid` | Required; references `products(id)` |
| `branch_id` | `uuid` | Required; references `branches(id)` |
| `employee_id` | `uuid` | Optional; references `employees(id)` |
| `quantity` | `integer` | Required; must be zero or greater |
| `counted_at` | `timestamptz` | Required; defaults to `now()` |
| `updated_at` | `timestamptz` | Required; defaults to `now()` and maintained by trigger |
| `created_at` | `timestamptz` | Required; defaults to `now()` |

Constraint:

- `chk_inventory_counts_quantity_non_negative`

Indexes:

- `idx_inventory_product` on (`product_id`)
- `idx_inventory_branch` on (`branch_id`)
- `idx_inventory_employee` on (`employee_id`)
- `idx_inventory_counted_at` on (`counted_at desc`)

Quantity may be zero but cannot be negative. There are no inventory sessions
and no `source` column. Current inventory is derived from the latest event.

### `audit_logs`

Stores append-only audit events.

| Column | Type | Rules |
|---|---|---|
| `id` | `uuid` | Primary key; defaults to `gen_random_uuid()` |
| `action` | `text` | Required |
| `entity_type` | `text` | Required |
| `entity_id` | `uuid` | Optional affected entity |
| `branch_id` | `uuid` | Optional; references `branches(id)` |
| `employee_id` | `uuid` | Optional; references `employees(id)` |
| `details` | `jsonb` | Optional structured event details |
| `created_at` | `timestamptz` | Required; defaults to `now()` |
| `device_info` | `jsonb` | Optional device metadata |
| `app_version` | `text` | Optional application version |

Indexes:

- `idx_audit_created_at` on (`created_at desc`)
- `idx_audit_employee` on (`employee_id`)
- `idx_audit_branch` on (`branch_id`)
- `idx_audit_entity` on (`entity_type`, `entity_id`)

This table has no `updated_at` column and no update trigger. Audit records are
append-only.

### `app_settings`

Stores system-wide settings.

| Column | Type | Rules |
|---|---|---|
| `key` | `text` | Primary key |
| `value` | `jsonb` | Required |
| `updated_at` | `timestamptz` | Required; defaults to `now()` and maintained by trigger |

Required seed values:

- `schema_version`: JSON string `"0.4.0"`
- `minimum_app_version`: JSON string `"0.4.0"`

## Updated-At Function and Triggers

`public.update_updated_at_column()` sets `new.updated_at = now()` before an
update.

The following triggers execute this function:

- `trg_branches_updated_at` on `branches`
- `trg_employees_updated_at` on `employees`
- `trg_employee_branches_updated_at` on `employee_branches`
- `trg_products_updated_at` on `products`
- `trg_inventory_counts_updated_at` on `inventory_counts`
- `trg_app_settings_updated_at` on `app_settings`

No trigger is created on `audit_logs`.

## `current_inventory` View

`public.current_inventory` returns the latest inventory count for each
product-and-branch pair. It includes:

- `inventory_count_id`
- `product_id`
- `barcode`
- `product_name`
- `image_path`
- `branch_id`
- `branch_name`
- `employee_id`
- `employee_name`
- `quantity`
- `counted_at`

Only active products and active branches are included. The latest event is
selected deterministically by:

1. `product_id`
2. `branch_id`
3. `counted_at desc`
4. `created_at desc`
5. `id desc`

Employee activity does not filter inventory history. The employee relationship
is optional, preserving counts that do not identify an employee.

## Supabase Storage

The `product-images` bucket is created with:

- ID: `product-images`
- Name: `product-images`
- Public access: `true`

The database stores only the corresponding object path in
`products.image_path`.

## Security Scope

RLS is intentionally not enabled. Authentication, users, roles, permissions,
and RLS will be introduced together in a later approved design. The present
schema must not be treated as authorization-ready until that work is complete.
